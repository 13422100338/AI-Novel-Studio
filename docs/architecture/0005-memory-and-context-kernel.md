# ADR 0005：长篇记忆与动态上下文内核

## 状态

已接受，并于 2026-07-03 在 Phase 4 实现。

## 决策

长篇记忆采用“原稿不动、结构化记录可追溯、模型结果先审查”的设计。正文仍由章节仓库保管；摘要、人物状态、读者知识、正典、叙事线索、文风规则和检索文档分别存储，不能用一次模型输出覆盖原稿或人工锁定记录。

所有记录使用稳定 UUID。章节标题、显示章号和卷名可以修改，但不作为关联键。旧项目通过幂等的 SQLite schema v2 迁移增加记忆表，不重写 Phase 1 的项目、卷、章和历史版本表。

## 时间边界

查询某一章的写前上下文时，只允许读取在该章之前成立的事件。时间顺序使用卷和章节的 `sort_index`，而不是显示章号。当前章和未来章中首次出现的事实、人物状态和线索动作默认不可见。

人物状态使用追加事件，不覆盖历史。角色可见边界整合进聚合人物状态卡，读者知识独立记录，防止把读者已知信息泄露给角色。相同边界上的同权冲突会全部返回并要求人工决定，程序不猜测哪条是真的。

## 权限与审查

记忆权限从高到低为用户确认、大纲、已审校、模型提取和推断。同权限内容冲突时不自动合并。模型提取的摘要、人物状态、正典、线索、读者知识和文风规则只生成 `REVIEW` 候选；晋升必须经过显式应用服务操作。

人工锁定的误导线索、开放问题、氛围暗示、正典和文风样章不能被模型修改。叙事线索保留类型和动作历史，`MISDIRECTION`、`OPEN_QUESTION` 与 `ATMOSPHERIC_HINT` 不会被当成事实矛盾自动修复。

## 摘要与失效

摘要层级为：

- L0：原始章节正文；
- L1：章节摘要；
- L2：情节段摘要；
- L3：卷摘要；
- L4：全书摘要。

摘要保存来源章节 ID、来源修订号和内容哈希。章节发生故事性修改时，依赖图把相关摘要、检索文档和 Context Manifest 标记为 `STALE`，但保留旧内容用于审计；纯错字修改可以由用户明确选择跳过记忆重建。

章节摘要使用固定的剧情概况、关键情节点、人物成长、连续性要点和细节摘录五段契约。剧情概况最多 1000 字，其余条目数量由模型按章节信息密度决定；细节摘录必须能在来源正文中逐字定位。伏笔、悬念、作者承诺和未决问题只进入叙事线索账本，不在章节摘要中重复。模型违反这些语义约束时，候选不会保存。

## 检索

历史检索使用 SQLite FTS5 `trigram`，不依赖外部向量数据库。结果综合全文匹配、人物参与、人工固定权重、章节距离、审查状态和过期惩罚，并以稳定 ID 解决同分排序。检索结果只提供证据，不授权删除未命中的原稿。

## Token 预算与 Context Manifest

用户设置输出 Token 上限，程序根据模型上下文窗口和安全余量计算输入预算：

```text
输入预算 = 上下文窗口 - 用户输出上限 - 安全余量
```

如果用户设置超过模型能力，程序返回明确错误，不静默缩减。`ContextBuilder` 按优先级选择完整证据块：必需约束必须完整保留；近期章节全文优先；较早内容可使用明确的摘要回退；无法容纳的整块内容被省略并记录原因。程序不从段落中间截断文本。

每次组装都会生成 JSON Context Manifest，记录选中和省略的块、来源 ID、章节修订、哈希、选择理由、是否使用摘要、估算 Token 和预算。清单保存在 `.ai_pipeline/manifests/`，数据库只保存索引和依赖状态。

## 模型与 UI 边界

记忆提取通过统一模型网关的 `MEMORY_EXTRACTION` 路由，先经过顶层 JSON 合约，再验证每个嵌套候选的枚举和必填字段。无效输出不会保存，也不会靠程序猜测修复。

记忆页面通过 `MemoryWorkspaceService` 绑定。UI 只发出加载、人工编辑和显式晋升意图；锁定保护和乐观修订检查在应用边界执行，实际存储由网关/仓库实现。离线演示数据仍可运行，但不会写入项目数据库。

## 阶段边界

Phase 4 不生成章节正文，不冻结 Brief，不执行自动修复，也不开放 Agent 工具循环。Phase 5 负责正式正文流水线与检查点，Phase 6 负责审校和有界修复，Phase 7 才会在只读工具与权限预算下引入 Agent 调用。

## 人物聚合状态卡

剧情商讨与正文生成共用 CharacterCardContextService 渲染人物卡候选，避免两条流水线使用不同字段或提示格式。共享卡包含长期特征、当前心理与目标、动机、关系、近期活动和最多八次更早心路变化；最新状态不会在“当前状态”和“过往心路”中重复。两条流水线仍可依据各自 Token 预算省略候选，但不会自行重新拼装另一种人物状态文本。

人物状态事件仍是持久化事实源，继续按章节追加，以保留回看任意时间点和审计历史的能力；不把已有项目迁移为每人仅一行的覆盖式记录。

旧版人物知识事件继续保留在数据库与仓储层，作为兼容和回滚数据，但不再由稿件整理流程生成、在记忆库展示或注入正文与 Brief 上下文。当前人物相关上下文统一来自聚合人物状态卡。读者知识事件继续作为可追溯事实源，应用层按章节边界聚合成一张大白话摘要卡；Brief、剧情商讨和正文生成共享同一摘要文本。人工保存摘要时写入一个高权威覆盖事件，不删除旧事件。

旧正典事实继续按单条记录持久化，应用层通过 CanonCardContextService 聚合为世界观、人物身份背景、重要物品与能力、组织与团队四张上下文卡。旧记录没有显式分类时使用确定性的标题关键词归类，未命中者进入世界观；用户可以在记忆工作区为单条事实保存显式分类，人工分类优先于关键词推断，但不会改写事实标题或正文。同标题、同最高权威但详情不同的记录必须显示为冲突，不得自动选真。剧情商讨和正文生成共用四卡文本。

应用层对 UI 和模型提供“每个人物一张状态卡”的时间有界只读视图。卡片的当前值取目标章节之前最新的 `APPROVED` 或 `LOCKED` 状态，过往心路按卷章顺序聚合已审查事件，并排除待审查、已删除章节、当前章和未来章数据。同一时间边界存在多个有效状态时继续报告冲突，不由程序猜测合并。
