# ADR 0020: Evolve HistoryRetriever through route-aware hybrid recall

## Status

Accepted for backend Phase 3 on 2026-07-19.

## Context

`HistoryRetriever` and `SearchRepository` already form the production history-search path used by
Brief compilation and generation memory. The repository previously issued one quoted FTS5 query.
That worked when the task repeated an indexed phrase, but it missed relevant history when the same
concept was expressed with separated keywords. Participant matching affected reranking only; it
could not recall a subject's history when the lexical query returned no candidates.

The backend plan calls for Hybrid Retrieval, while the architecture clarification requires the
existing retriever and repository to evolve instead of creating a second search system or truth
source.

## Decision

`SearchRepository.search_rows` now performs three bounded recall routes behind its existing API:

1. `EXACT_PHRASE` keeps the quoted FTS5 query;
2. `KEYWORD` derives a parameterized OR query from bounded ASCII terms and CJK trigrams;
3. `SUBJECT` recalls approved or locked rows whose stored participant IDs match the requested
   subjects.

Every route applies the same chapter boundary and review-state filter. Current and future chapters
remain ineligible. Subject-only candidates have no fabricated BM25 rank, so `HistoryRetriever`
assigns them a zero lexical score and ranks them with the existing participant, pin, recency, and
staleness signals.

Rows are merged by the existing memory-document ID before final reranking. The merged hit records
all contributing routes in deterministic order, which makes recall behavior inspectable without a
new persistence model. Search input is bounded, FTS terms are generated by deterministic code, and
all values remain SQLite parameters. The lexical score also preserves a base score for any FTS
match while increasing with SQLite's negative BM25 strength; stronger matches no longer receive a
slightly lower score because of the old absolute-value transform.

## Consequences

- Brief and generation callers gain broader recall without changing their storage or pipeline
  ownership.
- One document cannot consume multiple result slots merely because several routes found it.
- Review candidates and future chapters cannot enter through the new subject route.
- No schema migration, vector database, model call, manuscript change, or second retriever is
  introduced.
- Embedding recall remains a later route behind the same repository/retriever contract. Its vector
  representation, invalidation policy, and deterministic fallback must be designed and tested
  separately before it is enabled.
