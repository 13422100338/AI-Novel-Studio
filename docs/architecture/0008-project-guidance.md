# ADR 0008：项目级小说最高提示

## 状态

已接受，于 2026-07-15 实施持久化、人工编辑、只读访问和双模型输入接入。

## 背景

整部小说需要一份稳定的最高创作指令，记录基本目的、主题、写作视角和不可违背的总原则。
它的权威高于章节要求和动态记忆，并且不能被记忆提取模型、剧情商讨模型或正文模型自动修改。

`project.json` 按 ADR 0002 只保存项目身份和格式信息，不适合保存创作正文。现有记忆候选则具有模型提取、审查和晋升生命周期，也不符合“仅人工维护”的约束。

## 决策

- 在 SQLite schema v8 增加每项目一行的 `project_guidance` 表；
- 使用修订号进行乐观并发检查，旧窗口不能覆盖新修改；
- 只提供 `save_manual` 人工写入口；模型侧以后只能调用只读的 `read_highest_system_prompt`；
- 在记忆库首个页签提供人工编辑器，不提供 AI 整理或自动覆盖按钮；
- 正文生成把非空最高提示作为必需的 `PROJECT_GUIDANCE` 系统上下文块，
  并在 Context Manifest 中记录项目 ID、提示修订号和内容哈希；
- 剧情商讨在历史聊天和前文摘要之前注入同一份只读提示与最近三章全文；
  章节按全书顺序排列，且不包含当前章。即使摘要为空或摘要预算不足，
  也不会静默省略最高提示或这三章全文；
- 分层摘要使用扣除上述共享前缀后的剩余预算；正文侧可独立选择摘要节点，
  不会重复为最高提示或最近章节扣减摘要预算；
- 自动压缩摘要若覆盖最近三章中的任一章节，则不再加入本次模型输入；
  最近内容以全文为准，压缩摘要只负责更早的剧情，避免同一事件重复占用 Token；
- AI 参考内容窗口显示每个来源的修订号，便于确认本次生成实际读取的版本。

## 后果

最高提示与模型候选记忆在存储和权限上明确分离。该能力由 schema v8 引入；旧项目打开时会幂等升级到当前 schema，原稿和既有记忆不被重写。两个模型读取相同的人工最高提示，但模型没有对应写入口。正文输入预算不足以容纳该必需提示时会明确失败，而不是在用户不知情时省略。
